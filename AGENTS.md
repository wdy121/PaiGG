# PaiGG / 配局

## 项目目标
- 做一个面向中国大陆用户的 H5 验证页，验证 `CS2 / 瓦罗兰特玩家是否愿意使用互补匹配来找队友`
- 当前阶段目标只有一个：让用户完成资料填写并进入匹配池

## 当前版本边界
- 这是 `H5 验证版`，不是完整社交产品
- 用户前台：`移动端优先`，PC 仅做基础兼容
- 流程：填写资料 -> 进入匹配池 -> 人工在飞书撮合 -> 后续通知用户查看结果
- 后台数据：前端调用后端接口，由 `API Route / 后端接口` 中转写入飞书多维表格
- 手机号与验证码：`只做 UI 占位`，当前不接真实短信能力
- 当前不做登录，前面按游客态填写，入池前再展示手机号验证 UI

## 页面范围
- 只做验证版必要页面与状态
- 首屏：清晰表达价值、支持 CTA
- 分步骤资料页：像创建游戏名片，不像长问卷
- 入池成功/等待页
- 匹配结果页仅展示：
  - 对方昵称
  - 段位
  - 在线时间
  - 主位置/打法
  - 匹配理由
  - 复制游戏 ID

## 设计风格
- 产品名：`配局`
- 副标题：`为 CS2 / 瓦罗兰特玩家匹配更互补的队友`
- 风格：偏瓦罗兰特式的干净竞技感
- 关键词：`深色、干净、卡片化、轻电竞、克制、有产品感`
- 严禁：`问卷感、营销页感、恋爱社交感、过度赛博朋克`

## 交互原则
- 一屏一个核心决策
- 多用卡片选择，少用输入框
- 分步骤优先，不做超长单页表单
- 允许部分步骤跳过，但核心转化链路要完整
- 文案简短直接，优先降低填写负担

## 组件与实现原则
- 技术栈默认：`Next.js + TypeScript + Tailwind CSS`
- 组件优先服务于验证效率，不提前抽象复杂通用层
- 先有清晰的页面骨架，再补局部组件
- 表单组件优先支持：
  - 卡片单选/多选
  - 分步流转
  - 移动端安全区与点击区
- 样式优先用 Tailwind，就近表达；仅在重复明显时再抽离
- 任何新增依赖都要轻量且必要，避免重型 UI / 状态管理库

## 明确禁止
- 不擅自扩展为聊天、评论区、社区广场、会员、付费等功能
- 不提前接真实短信、真实登录、复杂推荐算法
- 不为了“像产品”而增加无验证价值的页面
- 不写成调研问卷或传统营销落地页
- 不在未确认需求时顺手改下一轮内容

## 数据与接口约定
- 前端不直连飞书；统一经 `API Route / 服务端接口` 中转
- 提交结构要按“可被人工撮合使用”设计，字段清晰、可枚举、便于落表
- 在真实后端未接入前，允许使用 mock 数据，但要保持接口边界稳定

## 协作约定
- 每轮只处理当前任务范围
- 改动保持小而清晰，方便后续迭代
- 输出时说明：
  - 改了哪些文件
  - 为什么这样改
  - 如何本地验证
